## Context

当前仓库为空，需要从零建立一个可扩展的 Flutter 移动端工程。该工程需要同时服务 iOS、Android 与鸿蒙三类目标平台，因此除了标准 Flutter 应用骨架，还需要提前约定平台目录分层、构建入口、依赖接入方式，以及对鸿蒙适配的集成边界。由于鸿蒙对 Flutter 的支持通常依赖特定适配方案或宿主工程约束，本次设计会把“共享 Flutter 层”和“平台宿主层”明确分离，避免后续接入业务功能时频繁调整基础结构。

## Goals / Non-Goals

**Goals:**
- 创建一个标准 Flutter 应用基座，作为三端共享业务代码的宿主。
- 提供 iOS、Android 的标准运行与构建基础能力。
- 为鸿蒙预留清晰的接入路径、目录和说明，确保项目结构对鸿蒙适配友好。
- 提供最小可运行页面和必要文档，让团队可以立即开始后续开发。

**Non-Goals:**
- 不在本次变更中实现具体业务页面、登录、网络层或状态管理方案。
- 不在本次变更中完成正式签名、上架参数或生产环境 CI/CD。
- 不承诺在所有本地机器上自动安装 Flutter、Xcode、Android Studio 或鸿蒙工具链。

## Decisions

### 使用标准 Flutter application 模板作为共享代码基础
选择标准 Flutter 应用工程作为起点，而不是手动拼装目录。这样可以保留 Flutter 官方工具链兼容性，降低未来升级 SDK、接入插件和执行 `flutter build` 时的维护成本。

备选方案：
- 手动创建 Dart 工程并自行组织平台目录：灵活，但会偏离 Flutter 官方约定，增加维护成本。
- 直接以现成业务模板起步：速度快，但会把与业务无关的复杂度提前引入空仓库。

### 共享层与平台宿主层分离
共享业务代码放在 Flutter/Dart 层，iOS、Android、鸿蒙各自保留独立宿主工程与配置文件。这样可以在不破坏共享层结构的前提下处理平台权限、签名、构建参数和插件差异。

备选方案：
- 将平台特定逻辑混入共享层：实现快，但会使未来适配和排错变得困难。

### 鸿蒙采用“兼容性优先”的接入策略
考虑到鸿蒙生态中的 Flutter 适配实现与官方移动平台不同，本次变更优先输出清晰的集成入口、目录规划和文档约束，而不是假设所有插件都能零成本直连。实施时应优先选用已验证的鸿蒙 Flutter 适配方案，并把平台相关桥接代码限制在宿主层。

备选方案：
- 直接承诺与所有 Flutter 插件完全兼容：风险过高，且缺乏当前仓库上下文支撑。

### 提供最小可运行首页用于三端验证
先构建一个简单、明确的首页而不是空白壳工程，用于验证共享入口、路由骨架和平台启动链路已打通。这能让后续 `/opsx:apply` 的实现具备可观测验证目标。

备选方案：
- 仅生成默认计数器示例：可运行，但不利于形成团队自己的应用骨架。

## Risks / Trade-offs

- [鸿蒙 Flutter 适配依赖具体方案与工具链版本] → 在实现中固定可用方案，并在文档中明确前置环境与限制。
- [三端一次性兼容会提升初始工程复杂度] → 保持首页和基础依赖最小化，避免过早引入业务框架。
- [平台构建要求不同，验证成本较高] → 把验证目标拆成共享层、iOS/Android、鸿蒙三部分，分别提供检查步骤。
- [空仓库缺少现有团队规范可参考] → 采用 Flutter 官方约定作为默认基线，并在文档中标出后续可扩展点。

## Migration Plan

1. 初始化 Flutter 工程并确认共享 Dart 层可以启动。
2. 补齐 iOS 与 Android 宿主工程配置，验证基础运行。
3. 集成鸿蒙所需宿主工程或适配目录，补充接入文档与构建入口。
4. 增加最小首页、资源目录和项目说明文件。
5. 逐平台执行基础验证，记录已知限制。

回滚策略：
如果某个平台接入阻塞，可以保留共享 Flutter 工程与已验证平台支持，把未完成的平台集成隔离在独立目录或文档中，避免破坏主工程骨架。

## Open Questions

- 当前计划采用哪一种鸿蒙 Flutter 适配发行版或宿主工程模板，需要在实施时最终确认。
- 是否需要在初始化阶段同步接入状态管理、路由和网络层框架，当前设计默认暂不引入。
- 是否需要在项目创建阶段一并配置 CI/CD，这里暂按后续独立变更处理。
