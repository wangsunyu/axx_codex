## 背景

当前仓库已经具备 Flutter 多端项目基座，但缺少真实业务登录流程。现有输入包括：

- 登录业务文档：[login/login.md](/Users/wsy/Desktop/code/iOS项目/安小信flutter-codex/login/login.md)
- 原 iOS 视觉参考图：[login1.PNG](/Users/wsy/Desktop/code/iOS项目/安小信flutter-codex/login/login1.PNG)、[login2.PNG](/Users/wsy/Desktop/code/iOS项目/安小信flutter-codex/login/login2.PNG)
- RSA 参考实现：[RSAEncryptor.m](/Users/wsy/Desktop/code/iOS项目/安小信flutter-codex/login/RSAEncryptor.m)

本次设计既要明确登录页如何在 Flutter 中实现，也要约束后续需求归档如何落地，避免需求散落在聊天记录中。

## 目标 / 非目标

**目标：**
- 设计一个与原 iOS 页面风格一致的双模式登录页。
- 明确账号密码登录与验证码登录的输入、校验、启用态、倒计时和提交规则。
- 明确 RSA 加密在 Flutter 侧的接入方式和接口调用边界。
- 建立统一的需求归档目录，便于项目后续按功能查阅需求。

**非目标：**
- 本次不定义注册、修改密码、找回密码完整流程，仅保留对应入口展示。
- 本次不扩展用户中心、权限体系或多角色登录规则。
- 本次不定义服务端接口协议变更，仅约束客户端如何对接现有接口。

## 关键设计决策

### 1. 登录页采用单页双模式切换，而不是拆成两个独立页面

原因：
- 原 iOS 截图体现的是同一视觉容器下的模式切换体验。
- 双模式共享顶部背景、Logo、登录按钮和底部辅助入口，单页实现更利于保持视觉一致和状态管理简单。

备选方案：
- 拆成两个页面分别承载账号密码登录与验证码登录。
- 放弃原因：切换成本更高，页面结构重复，且与参考图不一致。

### 2. 账号密码登录与验证码登录共用提交接口，但使用不同参数策略

原因：
- 需求已明确两种登录方式均调用 `zhxy-information/mobile/getToken.do`。
- 区别仅体现在 `password` 字段来源和 `loginType` 附加参数上，适合在同一认证服务中统一封装。

约定：
- 账号密码登录：`userCode` + RSA 加密后的 `password`
- 验证码登录：`userCode` + 原始验证码 `password` + `loginType=verificationCode`

### 3. RSA 加密行为保持与现有 iOS 参考实现一致

原因：
- 后端已基于现有 iOS 客户端完成联调，加密实现不宜自行发明新规则。
- 使用现有公钥进行 Flutter 侧加密可以减少接口兼容风险。

备选方案：
- 直接明文传输密码，由服务端自行处理。
- 放弃原因：与现有需求冲突，也会破坏既有接口契约。

### 4. 登录按钮和获取验证码按钮采用显式启用态控制

原因：
- 需求中已经给出明确阈值：账号密码登录要求账号和密码长度都 `>= 6`；验证码为 6 位；手机号需格式合法。
- 显式禁用和高亮状态能够直接对齐截图中的按钮视觉变化，并减少无效请求。

约定：
- 账号密码登录按钮：账号长度 `>= 6` 且密码长度 `>= 6` 时可点击。
- 验证码登录按钮：手机号合法且验证码长度为 `6` 时可点击。
- 获取验证码按钮：手机号合法且不在倒计时中时可点击。

### 5. 需求归档采用项目根目录固定文件夹维护

原因：
- 需求文档应脱离临时沟通上下文，成为长期可检索资产。
- 使用固定目录比散落在 `docs/` 或聊天记录中更直观，也便于后续按模块归档。

约定：
- 在项目根目录新增 `requirements/`
- 每个功能形成单独需求文件，文件名使用业务名称，例如 `requirements/登录需求.md`

## 风险 / 权衡

- [RSA 加密细节与 Flutter 实现不一致] → 先以现有 `RSAEncryptor` 公钥和填充方式为准，编码前增加一轮对照验证。
- [接口返回结构未在需求中完全写明] → 客户端展示层优先直接透传后端错误信息，成功态按 token 成功获取处理。
- [截图只覆盖主登录视图，没有补充异常态] → 异常、加载、倒计时中等状态采用与现有视觉风格一致的最小扩展实现。
- [需求归档格式后续可能扩展] → 首版先用 Markdown 固化基础模板，后续如有统一规范可平滑调整。
